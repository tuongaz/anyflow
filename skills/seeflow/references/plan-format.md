# Phase 4 plan-presentation template

This is the canonical format the `seeflow` skill renders to the user
at the end of Phase 4 (synthesis + schema validation), before writing any
files. It is the user's **last chance** to redirect before disk writes,
register, and end-to-end validation.

## Goals

- Show every change at a glance using the `+ / ~ / -` diff convention.
- Make Play and status placement legible without forcing the user to read
  JSON.
- List every file that will be written, so the user can spot accidental
  scope creep.
- End with a single, unambiguous prompt: `Reply 'go' to write, or describe
  what to change.`

## Diff prefixes

| Prefix | Meaning |
|---|---|
| `+` | NEW: node, connector, or file added vs the prior state. |
| `~` | MODIFIED: node or connector that existed and is being changed. |
| `-` | REMOVED: node, connector, or file dropped vs the prior state. |
| (no prefix) | Unchanged. Useful for connector lists where most edges stay the same; omit untouched nodes for brevity. |

For a **new** demo (no edit target), every node/connector/file gets `+`.

For an **edit** of an existing demo, prefixes annotate the delta:
`+` for additions, `~` for changes, `-` for removals. Untouched items can
be elided with a `(N unchanged)` line if the list is long.

## Template

```
## Plan for "<user prompt verbatim>"

Nodes (<N>)
  <prefix> <node-id>           [<nodeType>]    <Play / status summary>
  <prefix> <node-id>           [<nodeType>]    <Play / status summary>
  …

Connectors (<M>)
  <prefix> <source>  --<kind>→ <target>        <optional label>
  …

Files to write:
  <prefix> .seeflow/<slug>/seeflow.json
  <prefix> .seeflow/<slug>/scripts/<play-script>.ts
  <prefix> .seeflow/<slug>/scripts/<status-script>.ts
  <prefix> .seeflow/<slug>/state/.gitignore

<one-paragraph rationale: why these nodes / why these Plays / why these
statuses — pulled from the sub-agents' `oneNodeRationale` + `rationale`
fields. Keep it ≤ 4 lines.>

Reply 'go' to write, or describe what to change.
```

## Node summary column conventions

The summary column is free-text but should follow a stable shape so the
user can scan it:

- `Play: <trigger description>` — for nodes carrying a `playAction`.
- `status: <what is observable>` — for nodes carrying a `statusAction`.
- `Play + status` — for nodes carrying both.
- `(no play, no status)` — for nodes the audience sees but does not
  interact with directly (databases viewed via another node's status,
  decorative shapes).

Examples:

```
+ checkout-api          [playNode]    Play: POST /checkout (new cart fixture)
+ payments-service      [stateNode]   status: Stripe charge state polling
+ order-db              [stateNode]   status: orders row count + last state
~ fulfilment-worker     [stateNode]   (no play, no status) -- UNCHANGED
- email-sent            [stateNode]   REMOVED (was in existing demo)
+ user                  [iconNode]    (decorative)
```

## Connectors column conventions

Show direction with `--<kind>→`. The kind is one of the connector
discriminators (`http`, `event`, `queue`, `default`). The optional label
column reproduces the connector's `label` field (or `method url` for
http connectors, `eventName` for event connectors, `queueName` for queue
connectors).

```
  checkout-api  --http →  payments-service        POST /charge
  checkout-api  --event→  shipping-worker         order.created
  user          --default→ checkout-api           clicks checkout
```

## Files-to-write list

Always include `seeflow.json` and the `state/.gitignore`. List every script
file generated by the play-designer and status-designer (one file per
overlay). Examples:

```
+ .seeflow/checkout-flow/seeflow.json
+ .seeflow/checkout-flow/scripts/play-checkout.ts
+ .seeflow/checkout-flow/scripts/status-orders.ts
+ .seeflow/checkout-flow/scripts/status-shipments.ts
+ .seeflow/checkout-flow/state/.gitignore
```

For edits, mark file changes:

```
~ .seeflow/checkout-flow/seeflow.json
+ .seeflow/checkout-flow/scripts/status-payments.ts   (new)
~ .seeflow/checkout-flow/scripts/play-checkout.ts     (script body changed)
- .seeflow/checkout-flow/scripts/status-emails.ts     (no longer used)
```

## What NOT to include

- Raw JSON. The user should not need to read the schema to decide whether
  to say `go`.
- The full script bodies. Mention WHAT the scripts do in the rationale; if
  the user wants to see source, they can ask.
- Long lists of unchanged items in edit mode. Use `(N unchanged)` placeholders.
- `validationSafe`, `oneNodeRationale`, or any other sub-agent metadata
  field that won't reach disk. Surface the substance in the rationale
  paragraph; do not dump the raw fields.

## The closing prompt

The plan MUST end with this exact line:

```
Reply 'go' to write, or describe what to change.
```

This is load-bearing — the orchestrator waits for `go` (case-insensitive)
before transitioning to Phase 5. Any other reply is treated as a change
request and loops back to Phase 2 (for node/connector tweaks) or Phase 3
(for Play / status tweaks).
