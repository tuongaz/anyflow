## Codebase Patterns
- apps/cloud/ is a standard AWS CDK TypeScript project using `module: commonjs` in tsconfig (CDK requirement)
- Lambda handlers live in apps/cloud/lambda/api/index.ts and use @types/aws-lambda for APIGatewayProxyEventV2 types
- Run `bun install` inside apps/cloud/ separately (not covered by root bun.lock initially)
- CDK tsconfig must include both `bin/**/*.ts`, `lib/**/*.ts`, and `lambda/**/*.ts` to typecheck all code
- The apps/cloud/tsconfig.json uses `"module": "commonjs"` — different from the rest of the monorepo which uses ESM
- `HttpLambdaIntegration` is in stable `aws-cdk-lib/aws-apigatewayv2-integrations` (graduated from alpha package ~CDK 2.114); do NOT add `@aws-cdk/aws-apigatewayv2-integrations-alpha` as it stopped publishing at 2.114.1-alpha.0
- `NodejsFunction` (aws-cdk-lib/aws-lambda-nodejs) requires `esbuild` as a devDependency for bundling TypeScript Lambda handlers
- Lambda route dispatch: use `event.routeKey` (e.g. "POST /flows") to branch handlers in a single Lambda behind API Gateway HTTP API (v2)
- Use `randomUUID` from `node:crypto` (not global `crypto`) for UUID generation — explicit import works under commonjs tsconfig
- `fflate` (^0.8.2) is the right unzip library for Lambda: pure JS, bundles cleanly with esbuild, types included
- `@aws-sdk/client-s3` must be added to package.json for TypeScript types; esbuild marks it external so it uses the runtime SDK
- API Gateway HTTP API passes binary bodies base64-encoded; check `event.isBase64Encoded` before decoding
- ACM certificates for CloudFront must be in us-east-1; use a separate `SeeflowCertStack` with `crossRegionReferences: true` on both stacks
- `S3BucketOrigin.withOriginAccessControl()` (CDK 2.150+) auto-creates OAC + bucket policy for CloudFront → S3
- Extract API Gateway hostname for CloudFront HttpOrigin: `cdk.Fn.select(2, cdk.Fn.split('/', httpApi.apiEndpoint))`
- CloudFront `/api/*` behavior uses a CloudFront Function (JS_2_0) to strip the `/api` prefix before forwarding to API Gateway
