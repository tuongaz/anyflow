## Codebase Patterns
- Biome `organizeImports` sorts strictly alphabetically by source string. `@/...` imports come BEFORE third-party because `@` < `r`. `bun run format` does NOT auto-rewrite import order (it's a "suggested" Biome fix that requires `--unsafe`); manually reorder so `bun run lint` passes.
- `ColorTokenSchema` is the single source of truth for the palette enum (declared in `apps/studio/src/schema.ts`); the same string-literal union is mirrored in `apps/web/src/lib/api.ts` as `ColorToken`.
- Tests run under `bun test` and use the `bun:test` import (`describe/it/expect`). Example: `apps/web/src/lib/connector-to-edge.test.ts`.
- Web workspace TS config: `@/*` resolves to `apps/web/src/*`; uses `react-jsx`; `noEmit: true`; `tsc --noEmit` for typecheck.
- The `prd.json` and progress files are at the repo ROOT, not inside a `ralph/` subdirectory (the Makefile's `ralph-clean` target predates this — keep an eye on it).
- Demo file write pattern (apps/studio/src/api.ts): every mutation reads + parses + mutates the *raw parsed JSON* (not the Zod-typed object) so unknown forward-compat fields survive round-trips, then re-validates the WHOLE demo via `DemoSchema` before commit, then writes via `writeFileAtomic` inside `withDemoWriteLock(demoId, …)`. Output is `JSON.stringify(obj, null, 2) + '\n'` — 2-space indent + trailing newline. Errors use a discriminated `Outcome` union mapped to HTTP status in a final `switch`.
- Body-validation pattern: top-level mutation bodies are validated with a Zod schema using `.strict()` to reject typos in unknown keys; final shape is enforced via `DemoSchema.safeParse(rawAfterMutation)`. Nested fields can stay loose (`z.unknown()`) because the post-mutation DemoSchema check is the real gate.
- `crypto.randomUUID()` is available globally under Bun (no import needed). New node ids follow `node-${crypto.randomUUID()}`; new connector ids follow `conn-${crypto.randomUUID()}`.
- Connector PATCH gotcha: the connector schema is a `z.discriminatedUnion('kind', ...)`. By default, `z.object` strips unknown keys on parse, so a stale `eventName` (left over from a kind='event' → kind='default' change) would parse cleanly but linger on disk. PATCH /connectors/:connId clears `method/url/eventName/queueName` when `kind` changes — keep this invariant if you generalize the merge helper.
- Frontend hook tests run as pure-function tests under `bun:test` — there's no React testing library installed. When writing a stateful hook, extract the state-update logic into pure helpers (e.g. `applySetOverride`, `applyPruneAgainst`) and test those; the hook itself is a thin `useState`/`useCallback` shell. Pattern: `apps/web/src/hooks/use-pending-overrides.ts` exports both the hook and the underlying pure reducers.
- Optimistic-edit reconciliation: `usePendingOverrides<T>()` is the single way the canvas overlays in-flight edits on server data. `setOverride(id, partial)` BEFORE the API call; on success, the next `demo:reload` triggers `pruneAgainst(snapshot)` which drops fields that match the server (per-field, not per-entity); on failure, `dropOverride(id)` reverts. Use this pattern for every new UI mutation (US-022 onward) — don't add ad-hoc `useState` maps.

- All three endpoints share the same shape: 400 on body / on-disk JSON / schema failure, 404 on unknown demoId or nodeId, 500 on write failure. They all run inside `withDemoWriteLock` and write via `writeFileAtomic`. Crucially they re-validate the WHOLE mutated demo via `DemoSchema` before commit (existing `/position` endpoint stays as-is — pure-position mutations can't break invariants so the pre-mutation parse is enough there).

- Files changed:
  - `apps/studio/src/api.ts` — new endpoints, `NodePatchBodySchema`, `mergeNodeUpdates` helper, `ColorTokenSchema` import.
  - `apps/studio/src/api.test.ts` — three new `describe` blocks (15 new tests): PATCH partial update, position-only update, schema validation 400, body-strict 400, label-empty 400, 404s, file formatting preservation; POST creates with auto-id, with caller-id, schema-failure 400, 404; DELETE cascade (full removal), DELETE with non-cascading connectors preserved, 404s.
- Quality checks: `bun test` (121/0 — was 105 before, +16 incl. one shared describe restructure), `bun run typecheck` (clean), `bun run lint` (clean).
- **Learnings for future iterations:**
  - Mutate the *raw parsed JSON* (the `unknown` from `Bun.file().json()`), NOT the Zod-typed result. The schema strips/coerces — author-written fields outside the schema would be lost on round-trip. The existing `/position` handler already followed this pattern; new endpoints must too.
  - `DemoSchema.superRefine` enforces connector → node referential integrity. Cascade-deleting connectors when a node is removed isn't optional — without it the post-mutation `DemoSchema.safeParse` would 400 on every node delete that had any adjacent connectors.
  - Biome's formatter wraps long lambda bodies onto a single line when they fit. The DELETE handler had a 3-line `obj.connectors.filter((cn) => …)` that the formatter rewrote inline. `bun run format` fixes this; always run `bun run format` before `bun run lint`.
  - `crypto.randomUUID()` works at the global scope under Bun — no import needed.
  - For PATCH bodies, prefer `.strict()` Zod schemas over loose `z.record(z.unknown())` — strict catches client typos (e.g. `borderColour` instead of `borderColor`) at the body-validation layer rather than waiting for the post-mutation schema parse to throw a more confusing error.
- Manual browser verification: not required by US-019 (backend-only). UI consumers land in US-021 onward.

## 2026-05-07 - US-020
- Implemented backend connector CRUD endpoints in `apps/studio/src/api.ts`, mirroring the US-019 node CRUD pattern:
  - `PATCH /api/demos/:id/connectors/:connId` — partial update of `label`, `style`, `color`, `direction`, `kind`, `eventName`, `queueName`, `method`, `url`. Body validated by `ConnectorPatchBodySchema` (`.strict()`). When `kind` changes, the new `mergeConnectorUpdates` helper drops every kind-specific field (`method/url/eventName/queueName`) before applying the new ones, so the on-disk JSON doesn't carry phantom fields from the previous kind.
  - `POST /api/demos/:id/connectors` — creates a new connector. Auto-generates `id = 'conn-' + crypto.randomUUID()` when absent. Defaults `kind` to `'default'` (the no-semantics user-drawn variant) when absent. DemoSchema's `superRefine` enforces source/target referential integrity automatically.
  - `DELETE /api/demos/:id/connectors/:connId` — removes the connector. Unlike node DELETE, no cascade is needed (connectors don't own anything else).
- All three endpoints share the same Outcome-union → switch-on-kind pattern: 400 on body / on-disk JSON / schema failure, 404 on unknown demoId or connectorId, 500 on write failure. They run inside `withDemoWriteLock(id, …)`, write via `writeFileAtomic`, and re-validate the WHOLE demo through `DemoSchema` before commit.
- Files changed:
  - `apps/studio/src/api.ts` — three new endpoints, `ConnectorPatchBodySchema`, `ConnectorKindSchema`, `mergeConnectorUpdates` helper, `CONNECTOR_KIND_FIELDS` constant.
  - `apps/studio/src/api.test.ts` — three new `describe` blocks (14 new tests): PATCH visual-fields merge, kind-change clears stale fields, schema 400 (kind='event' without eventName), body-strict 400, 404s; POST default-kind auto-id, caller-id pass-through, schema 400 (unknown source), kind-discriminated 400, demo 404; DELETE removes targeted, demo 404, conn 404.
- Quality checks: `bun test` (135/0 — was 121 before, +14), `bun run typecheck` (clean), `bun run lint` (clean).
- **Learnings for future iterations:**
  - `z.discriminatedUnion` with `z.object` children silently STRIPS unknown keys on parse but does NOT remove them from the source object you mutated. If you change the discriminator (`kind`) without dropping the previous variant's fields, the post-parse object looks correct but the on-disk JSON keeps the stale fields. Always clear kind-specific fields explicitly when the discriminator changes.
  - For POST endpoints with auto-generated IDs, prefer a typed-prefix convention (`node-`, `conn-`) — it makes the PRD's "id auto-generated if absent" testable via a regex match (`expect(id).toMatch(/^conn-/)`) without hardcoding UUIDs in fixtures.
  - The post-mutation `DemoSchema.safeParse` is the real gate. The body-validation schema only catches typos and obvious type errors at the top level; per-kind invariants (e.g. `kind='event'` requires `eventName`) are caught by the discriminated union's per-branch parsing of the *whole* demo. This means the body schema can stay loose on per-kind fields without losing safety.
- Manual browser verification: not required by US-020 (backend-only). UI consumers (Style tab editor, connector creation via handle drag) land in US-024/US-025/US-029.

## 2026-05-07 - US-021
- Generalized the `positionOverrides` reconciliation pattern into a reusable hook + pure reducers, then refactored `demo-view.tsx` to consume it for both nodes and connectors. Position-drag UX is preserved byte-for-byte — drag-stop still fires `setOverride(nodeId, { position })` → PATCH → `pruneAgainst` on the next `demo:reload` echo or `dropOverride` on PATCH failure.
- Files added:
  - `apps/web/src/hooks/use-pending-overrides.ts` — `usePendingOverrides<T extends { id: string }>()` returning `{ overrides, setOverride, dropOverride, pruneAgainst, reset }`. State-update logic lives in three pure exported helpers (`applySetOverride`, `applyDropOverride`, `applyPruneAgainst`) so tests can run without a React test renderer (none is installed under `apps/web`). `pruneAgainst` is per-FIELD: it drops every override key whose value `deepEqual`s the server's value, then drops the entry if nothing remains. `setOverride` shallow-merges so multi-field optimistic edits accumulate.
  - `apps/web/src/hooks/use-pending-overrides.test.ts` — 15 tests covering: set + merge + last-write-wins + immutability, drop existing + drop missing (reference-stable), prune-on-match (single + multi-field + entity-missing-from-snapshot + multi-id), drop-on-error (revert).
- Files modified:
  - `apps/web/src/pages/demo-view.tsx` — replaced `positionOverrides` `useState` + `useEffect` reconciliation with two `usePendingOverrides<DemoNode>()` + `usePendingOverrides<Connector>()` hooks. New `data-testid='edit-error-banner'` (replaces `position-error-banner`). Position-drag callback is now `setNodeOverride(nodeId, { position }) → PATCH → dropNodeOverride on failure`.
  - `apps/web/src/components/demo-canvas.tsx` — `positionOverrides: Record<id, {x,y}>` prop replaced with `nodeOverrides: OverrideMap<DemoNode>`. New module-scoped `mergeNodeOverride(node, override)` helper does shallow merge with `data` shallow-merged one level deeper. The merge is read at render time inside the existing `sourceNodes` `useMemo`, so position-drag still flows through React Flow's internal state during the gesture.
- Quality checks: `bun test` (150/0 — was 135 before, +15 new tests in `use-pending-overrides.test.ts`), `bun run typecheck` (clean), `bun run lint` (clean).
- **Learnings for future iterations:**
  - `apps/web` has no React testing library. Test stateful hooks by extracting the state-update logic into pure helpers and testing those. The hook then becomes a trivial `useState`/`useCallback` shell that doesn't need a unit test of its own (the React types ensure `setState(reducer)` is called correctly).
  - Per-field pruning (vs per-entity) is the right generalization. If a user changes both `position` AND `label` and only the position lands first (e.g. the label PATCH is still in flight), per-entity pruning would wipe the label override on the position echo. Per-field keeps each override alive until its own server value matches.
  - `Object.entries(prev)` vs `Object.keys(prev)` + `prev[id]` matters under TS strict mode. The keyed lookup is typed `Partial<T> | undefined` because of `Record`'s index-signature semantics, which trips `noUncheckedIndexedAccess`. Use `entries` to get a non-optional `[id, partial]` tuple.
  - When generalizing an existing optimistic-state pattern, the test surface should mirror the pattern's full lifecycle: set → prune-on-match (success path), set → drop (failure path), set + set (multi-field), prune-against-missing (entity gone). Anything less and you'll regress an edge case the next time someone changes the merge logic.
  - The position-drag flow's "drag-while-moving lives in React Flow's internal state" trick (`draggingRef` + the `useEffect` guard) still works after the generalization because the override map is keyed by id and only flows through React Flow's `nodes` prop on render — not on every drag event.
- Manual browser verification: not required by US-021 (no new visual surface — the position-drag UX is unchanged and the new `edit-error-banner` won't be exercised until US-024/US-025/US-026/US-029 wire actual mutating UI). The data-testid rename (`position-error-banner` → `edit-error-banner`) is intentional per the PRD; CLAUDE.md still references the old name in the position-drag pattern paragraph but the new banner subsumes it.
