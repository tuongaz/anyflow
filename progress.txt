## Codebase Patterns
- Biome `organizeImports` sorts strictly alphabetically by source string. `@/...` imports come BEFORE third-party because `@` < `r`. `bun run format` does NOT auto-rewrite import order (it's a "suggested" Biome fix that requires `--unsafe`); manually reorder so `bun run lint` passes.
- `ColorTokenSchema` is the single source of truth for the palette enum (declared in `apps/studio/src/schema.ts`); the same string-literal union is mirrored in `apps/web/src/lib/api.ts` as `ColorToken`.
- Tests run under `bun test` and use the `bun:test` import (`describe/it/expect`). Example: `apps/web/src/lib/connector-to-edge.test.ts`.
- Web workspace TS config: `@/*` resolves to `apps/web/src/*`; uses `react-jsx`; `noEmit: true`; `tsc --noEmit` for typecheck.
- The `prd.json` and progress files are at the repo ROOT, not inside a `ralph/` subdirectory (the Makefile's `ralph-clean` target predates this ‚Äî keep an eye on it).
- Demo file write pattern (apps/studio/src/api.ts): every mutation reads + parses + mutates the *raw parsed JSON* (not the Zod-typed object) so unknown forward-compat fields survive round-trips, then re-validates the WHOLE demo via `DemoSchema` before commit, then writes via `writeFileAtomic` inside `withDemoWriteLock(demoId, ‚Ä¶)`. Output is `JSON.stringify(obj, null, 2) + '\n'` ‚Äî 2-space indent + trailing newline. Errors use a discriminated `Outcome` union mapped to HTTP status in a final `switch`.
- Body-validation pattern: top-level mutation bodies are validated with a Zod schema using `.strict()` to reject typos in unknown keys; final shape is enforced via `DemoSchema.safeParse(rawAfterMutation)`. Nested fields can stay loose (`z.unknown()`) because the post-mutation DemoSchema check is the real gate.
- `crypto.randomUUID()` is available globally under Bun (no import needed). New node ids follow `node-${crypto.randomUUID()}`; new connector ids follow `conn-${crypto.randomUUID()}`.
- Connector PATCH gotcha: the connector schema is a `z.discriminatedUnion('kind', ...)`. By default, `z.object` strips unknown keys on parse, so a stale `eventName` (left over from a kind='event' ‚Üí kind='default' change) would parse cleanly but linger on disk. PATCH /connectors/:connId clears `method/url/eventName/queueName` when `kind` changes ‚Äî keep this invariant if you generalize the merge helper.

They all run inside `withDemoWriteLock` and write via `writeFileAtomic`. Crucially they re-validate the WHOLE mutated demo via `DemoSchema` before commit (existing `/position` endpoint stays as-is ‚Äî pure-position mutations can't break invariants so the pre-mutation parse is enough there).

  - `apps/studio/src/api.ts` ‚Äî new endpoints, `NodePatchBodySchema`, `mergeNodeUpdates` helper, `ColorTokenSchema` import.

- **Learnings for future iterations:**
  - Mutate the *raw parsed JSON* (the `unknown` from `Bun.file().json()`), NOT the Zod-typed result. The schema strips/coerces ‚Äî author-written fields outside the schema would be lost on round-trip. The existing `/position` handler already followed this pattern; new endpoints must too.
  - `DemoSchema.superRefine` enforces connector ‚Üí node referential integrity. Cascade-deleting connectors when a node is removed isn't optional ‚Äî without it the post-mutation `DemoSchema.safeParse` would 400 on every node delete that had any adjacent connectors.
  - Biome's formatter wraps long lambda bodies onto a single line when they fit. The DELETE handler had a 3-line `obj.connectors.filter((cn) => ‚Ä¶)` that the formatter rewrote inline. `bun run format` fixes this; always run `bun run format` before `bun run lint`.
  - `crypto.randomUUID()` works at the global scope under Bun ‚Äî no import needed.
  - For PATCH bodies, prefer `.strict()` Zod schemas over loose `z.record(z.unknown())` ‚Äî strict catches client typos (e.g. `borderColour` instead of `borderColor`) at the body-validation layer rather than waiting for the post-mutation schema parse to throw a more confusing error.
- Manual browser verification: not required by US-019 (backend-only). UI consumers land in US-021 onward.

## 2026-05-07 - US-020
- Implemented backend connector CRUD endpoints in `apps/studio/src/api.ts`, mirroring the US-019 node CRUD pattern:
  - `PATCH /api/demos/:id/connectors/:connId` ‚Äî partial update of `label`, `style`, `color`, `direction`, `kind`, `eventName`, `queueName`, `method`, `url`. Body validated by `ConnectorPatchBodySchema` (`.strict()`). When `kind` changes, the new `mergeConnectorUpdates` helper drops every kind-specific field (`method/url/eventName/queueName`) before applying the new ones, so the on-disk JSON doesn't carry phantom fields from the previous kind.
  - `POST /api/demos/:id/connectors` ‚Äî creates a new connector. Auto-generates `id = 'conn-' + crypto.randomUUID()` when absent. Defaults `kind` to `'default'` (the no-semantics user-drawn variant) when absent. DemoSchema's `superRefine` enforces source/target referential integrity automatically.
  - `DELETE /api/demos/:id/connectors/:connId` ‚Äî removes the connector. Unlike node DELETE, no cascade is needed (connectors don't own anything else).
- All three endpoints share the same Outcome-union ‚Üí switch-on-kind pattern: 400 on body / on-disk JSON / schema failure, 404 on unknown demoId or connectorId, 500 on write failure. They run inside `withDemoWriteLock(id, ‚Ä¶)`, write via `writeFileAtomic`, and re-validate the WHOLE demo through `DemoSchema` before commit.
- Files changed:
  - `apps/studio/src/api.ts` ‚Äî three new endpoints, `ConnectorPatchBodySchema`, `ConnectorKindSchema`, `mergeConnectorUpdates` helper, `CONNECTOR_KIND_FIELDS` constant.
  - `apps/studio/src/api.test.ts` ‚Äî three new `describe` blocks (14 new tests): PATCH visual-fields merge, kind-change clears stale fields, schema 400 (kind='event' without eventName), body-strict 400, 404s; POST default-kind auto-id, caller-id pass-through, schema 400 (unknown source), kind-discriminated 400, demo 404; DELETE removes targeted, demo 404, conn 404.
- Quality checks: `bun test` (135/0 ‚Äî was 121 before, +14), `bun run typecheck` (clean), `bun run lint` (clean).
- **Learnings for future iterations:**
  - `z.discriminatedUnion` with `z.object` children silently STRIPS unknown keys on parse but does NOT remove them from the source object you mutated. If you change the discriminator (`kind`) without dropping the previous variant's fields, the post-parse object looks correct but the on-disk JSON keeps the stale fields. Always clear kind-specific fields explicitly when the discriminator changes.
  - For POST endpoints with auto-generated IDs, prefer a typed-prefix convention (`node-`, `conn-`) ‚Äî it makes the PRD's "id auto-generated if absent" testable via a regex match (`expect(id).toMatch(/^conn-/)`) without hardcoding UUIDs in fixtures.
  - The post-mutation `DemoSchema.safeParse` is the real gate. The body-validation schema only catches typos and obvious type errors at the top level; per-kind invariants (e.g. `kind='event'` requires `eventName`) are caught by the discriminated union's per-branch parsing of the *whole* demo. This means the body schema can stay loose on per-kind fields without losing safety.
- Manual browser verification: not required by US-020 (backend-only). UI consumers (Style tab editor, connector creation via handle drag) land in US-024/US-025/US-029.
