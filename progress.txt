## Codebase Patterns
- Biome `organizeImports` sorts strictly alphabetically by source string. `@/...` imports come BEFORE third-party because `@` < `r`. `bun run format` does NOT auto-rewrite import order (it's a "suggested" Biome fix that requires `--unsafe`); manually reorder so `bun run lint` passes.
- `ColorTokenSchema` is the single source of truth for the palette enum (declared in `apps/studio/src/schema.ts`); the same string-literal union is mirrored in `apps/web/src/lib/api.ts` as `ColorToken`.
- Tests run under `bun test` and use the `bun:test` import (`describe/it/expect`). Example: `apps/web/src/lib/connector-to-edge.test.ts`.
- Web workspace TS config: `@/*` resolves to `apps/web/src/*`; uses `react-jsx`; `noEmit: true`; `tsc --noEmit` for typecheck.
- The `prd.json` and progress files are at the repo ROOT, not inside a `ralph/` subdirectory (the Makefile's `ralph-clean` target predates this — keep an eye on it).
- Demo file write pattern (apps/studio/src/api.ts): every mutation reads + parses + mutates the *raw parsed JSON* (not the Zod-typed object) so unknown forward-compat fields survive round-trips, then re-validates the WHOLE demo via `DemoSchema` before commit, then writes via `writeFileAtomic` inside `withDemoWriteLock(demoId, …)`. Output is `JSON.stringify(obj, null, 2) + '\n'` — 2-space indent + trailing newline. Errors use a discriminated `Outcome` union mapped to HTTP status in a final `switch`.
- Body-validation pattern: top-level mutation bodies are validated with a Zod schema using `.strict()` to reject typos in unknown keys; final shape is enforced via `DemoSchema.safeParse(rawAfterMutation)`. Nested fields can stay loose (`z.unknown()`) because the post-mutation DemoSchema check is the real gate.
- `crypto.randomUUID()` is available globally under Bun (no import needed). New node ids follow `node-${crypto.randomUUID()}`; new connector ids follow `conn-${crypto.randomUUID()}`.
- Connector PATCH gotcha: the connector schema is a `z.discriminatedUnion('kind', ...)`. By default, `z.object` strips unknown keys on parse, so a stale `eventName` (left over from a kind='event' → kind='default' change) would parse cleanly but linger on disk. PATCH /connectors/:connId clears `method/url/eventName/queueName` when `kind` changes — keep this invariant if you generalize the merge helper.
- Frontend hook tests run as pure-function tests under `bun:test` — there's no React testing library installed. When writing a stateful hook, extract the state-update logic into pure helpers (e.g. `applySetOverride`, `applyPruneAgainst`) and test those; the hook itself is a thin `useState`/`useCallback` shell. Pattern: `apps/web/src/hooks/use-pending-overrides.ts` exports both the hook and the underlying pure reducers.
- Optimistic-edit reconciliation: `usePendingOverrides<T>()` is the single way the canvas overlays in-flight edits on server data. `setOverride(id, partial)` BEFORE the API call; on success, the next `demo:reload` triggers `pruneAgainst(snapshot)` which drops fields that match the server (per-field, not per-entity); on failure, `dropOverride(id)` reverts. Use this pattern for every new UI mutation (US-022 onward) — don't add ad-hoc `useState` maps.
- Frontend type sync: `apps/web/src/lib/api.ts` mirrors the studio's `apps/studio/src/schema.ts` Demo types as plain TS interfaces (no zod on the frontend). When schema additions land server-side, mirror them in `api.ts`. Discriminated unions in `api.ts` use inline-typed branches so the variant-name TS types don't collide with the React component names of the same name (e.g. `PlayNode` is a component in `nodes/play-node.tsx` — don't also export a `PlayNode` type from api.ts; build the union from anonymous branches instead).
- Discriminated union spread gotcha: `mergeNodeOverride(node: DemoNode, override: Partial<DemoNode>)` in `apps/web/src/components/demo-canvas.tsx` returns a `DemoNode`, but TS can't preserve the variant through `{...node, ...override, data}` because `Partial<DemoNode>` flattens `data` to a union of partials. The override is keyed by the entity id so it's always the same variant — safe to cast at the boundary (`as DemoNode`). Future merges of partial-of-discriminated-union should follow the same trick.
- Studio schema is loaded once at process start. After modifying `apps/studio/src/schema.ts`, restart the studio (`bun run apps/studio/src/cli.ts stop` then `... start --daemon`) before browser-verifying. The demo files will round-trip through the new schema on the next `demo:reload`. Live-reload only covers the frontend bundle (Vite HMR); backend changes need a restart.
- NodeResizer wiring (US-023): each custom node component renders `<NodeResizer isVisible={selected && !!data.onResize} minWidth={80} minHeight={40} onResizeStart={…} onResizeEnd={…}/>` inside its outer card. `data.onResize` + `data.setResizing` are injected by `DemoCanvas` via the `data` field of the React Flow Node spec (same channel as `onPlay`). On resize-stop, the node fires `data.onResize(id, {width,height})` which lands in `DemoView.onNodeResize` → optimistic `setNodeOverride(id, { data: { width, height } })` → `updateNode(demoId, id, dims)` PATCH. Inner cards switch to `h-full w-full` while resizing OR when `data.width/data.height` are set; otherwise they keep natural sizing (`w-[260px]` for play/state, per-shape inline dims for shape). DemoCanvas blocks the `setRfNodes(sourceNodes)` echo while either `draggingRef` OR `resizingRef` is true (mirrors the position-drag pattern).
- Inspector selection state (US-024): `DemoView` owns BOTH `selectedId` (node) and `selectedConnectorId` (connector) as separate `useState`s, but they're mutually exclusive — `onSelectNode(id)` clears the connector and vice versa. `DemoCanvas` accepts both and wires React Flow's `onNodeClick` + `onEdgeClick`; `onPaneClick` clears both. Edges get `selected: true` when their id matches `selectedConnectorId` (assigned to the `DerivedEdge` after `connectorToEdge` returns). `DetailPanel` accepts both `node` and `connector` props; the Sheet is open when either is set, and switching between them keeps the Sheet mounted.
- Tabbed sidebar (US-024): `DetailPanel` wraps both node and connector content in `<Tabs value={tab}>` (Detail + Style). `tab` lives in a local `useState<TabKey>` and is reset to `'detail'` whenever the selected entity id changes (`useEffect` keyed on `selectionId = node?.id ?? connector?.id`). This means tab choice persists across SSE-driven re-renders that keep the same selection, but a brand-new selection always opens on Detail. Style tab contents are intentional placeholders for now — US-025 fills them in. New shadcn primitive at `apps/web/src/components/ui/tabs.tsx` (depends on `@radix-ui/react-tabs`, added via `bun add` in the `apps/web` workspace).

  - When extending the position-drag-blocks-echo pattern (the `draggingRef` in DemoCanvas), use a parallel `resizingRef` AND let nodes toggle it via a `data.setResizing` callback. Avoid triggering React state changes for "is this node resizing?" at the canvas level — re-renders during a gesture would force React Flow to re-do its measurement and worsen the UX.
  - Biome's `organizeImports` puts `type NodeProps` before `NodeResizer` in `from '@xyflow/react'` because alphabetical sort is by the imported name (ignoring the `type` keyword). Run `bun run lint` to surface this; the auto-fix is "suggested" so manual reorder is required.
  - The `data` channel (the `Record<string, unknown>` slot of a React Flow Node) is the canonical place to inject per-node callbacks (`onPlay`, `onResize`, `setResizing`). Don't add a parallel context or prop drilling — the existing pattern (`onPlay` since US-007) already proves it works for SSE-tick stability.
  - `updateNode(demoId, nodeId, patch)` is the generic frontend PATCH helper for `/api/demos/:id/nodes/:nodeId`. Existing `updateNodePosition` stays as the position-only fast path (per the US-019 acceptance criteria). Future UI mutations (label, colors, detail) should funnel through `updateNode` rather than adding a new helper per field.
