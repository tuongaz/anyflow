## Codebase Patterns
- Biome `organizeImports` sorts strictly alphabetically by source string. `@/...` imports come BEFORE third-party because `@` < `r`. `bun run format` does NOT auto-rewrite import order (it's a "suggested" Biome fix that requires `--unsafe`); manually reorder so `bun run lint` passes.
- `ColorTokenSchema` is the single source of truth for the palette enum (declared in `apps/studio/src/schema.ts`); the same string-literal union is mirrored in `apps/web/src/lib/api.ts` as `ColorToken`.
- Tests run under `bun test` and use the `bun:test` import (`describe/it/expect`). Example: `apps/web/src/lib/connector-to-edge.test.ts`.
- Web workspace TS config: `@/*` resolves to `apps/web/src/*`; uses `react-jsx`; `noEmit: true`; `tsc --noEmit` for typecheck.
- The `prd.json` and progress files are at the repo ROOT, not inside a `ralph/` subdirectory (the Makefile's `ralph-clean` target predates this â€” keep an eye on it).
- Demo file write pattern (apps/studio/src/api.ts): every mutation reads + parses + mutates the *raw parsed JSON* (not the Zod-typed object) so unknown forward-compat fields survive round-trips, then re-validates the WHOLE demo via `DemoSchema` before commit, then writes via `writeFileAtomic` inside `withDemoWriteLock(demoId, â€¦)`. Output is `JSON.stringify(obj, null, 2) + '\n'` â€” 2-space indent + trailing newline. Errors use a discriminated `Outcome` union mapped to HTTP status in a final `switch`.
- Body-validation pattern: top-level mutation bodies are validated with a Zod schema using `.strict()` to reject typos in unknown keys; final shape is enforced via `DemoSchema.safeParse(rawAfterMutation)`. Nested fields can stay loose (`z.unknown()`) because the post-mutation DemoSchema check is the real gate.
- `crypto.randomUUID()` is available globally under Bun (no import needed). New node ids follow `node-${crypto.randomUUID()}`; new connector ids follow `conn-${crypto.randomUUID()}`.
- Connector PATCH gotcha: the connector schema is a `z.discriminatedUnion('kind', ...)`. By default, `z.object` strips unknown keys on parse, so a stale `eventName` (left over from a kind='event' â†’ kind='default' change) would parse cleanly but linger on disk. PATCH /connectors/:connId clears `method/url/eventName/queueName` when `kind` changes â€” keep this invariant if you generalize the merge helper.
- Frontend hook tests run as pure-function tests under `bun:test` â€” there's no React testing library installed. When writing a stateful hook, extract the state-update logic into pure helpers (e.g. `applySetOverride`, `applyPruneAgainst`) and test those; the hook itself is a thin `useState`/`useCallback` shell. Pattern: `apps/web/src/hooks/use-pending-overrides.ts` exports both the hook and the underlying pure reducers.
- Optimistic-edit reconciliation: `usePendingOverrides<T>()` is the single way the canvas overlays in-flight edits on server data. `setOverride(id, partial)` BEFORE the API call; on success, the next `demo:reload` triggers `pruneAgainst(snapshot)` which drops fields that match the server (per-field, not per-entity); on failure, `dropOverride(id)` reverts. Use this pattern for every new UI mutation (US-022 onward) â€” don't add ad-hoc `useState` maps.
- Frontend type sync: `apps/web/src/lib/api.ts` mirrors the studio's `apps/studio/src/schema.ts` Demo types as plain TS interfaces (no zod on the frontend). When schema additions land server-side, mirror them in `api.ts`. Discriminated unions in `api.ts` use inline-typed branches so the variant-name TS types don't collide with the React component names of the same name (e.g. `PlayNode` is a component in `nodes/play-node.tsx` â€” don't also export a `PlayNode` type from api.ts; build the union from anonymous branches instead).
- Discriminated union spread gotcha: `mergeNodeOverride(node: DemoNode, override: Partial<DemoNode>)` in `apps/web/src/components/demo-canvas.tsx` returns a `DemoNode`, but TS can't preserve the variant through `{...node, ...override, data}` because `Partial<DemoNode>` flattens `data` to a union of partials. The override is keyed by the entity id so it's always the same variant â€” safe to cast at the boundary (`as DemoNode`). Future merges of partial-of-discriminated-union should follow the same trick.
- Studio schema is loaded once at process start. After modifying `apps/studio/src/schema.ts`, restart the studio (`bun run apps/studio/src/cli.ts stop` then `... start --daemon`) before browser-verifying. The demo files will round-trip through the new schema on the next `demo:reload`. Live-reload only covers the frontend bundle (Vite HMR); backend changes need a restart.
